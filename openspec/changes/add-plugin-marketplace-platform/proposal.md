## Why

CRMFRANKLIN is being open-sourced and needs a third-party ecosystem on top of the open-source core. The target shape is "Open Core + Centralized Marketplace": the open-source CRM instance is the **data plane**, and a separate centralized SaaS (closed-source, out of scope here) is the **control plane** for catalog, distribution, licensing, and review. To support plugins/mods/UI/themes on the open-source core, the instance needs an in-core extension framework, machine-to-machine plugin authorization, and a disableable marketplace client. The current architecture cannot support any of this:

- AI capability registration is hardcoded and closed. `ChatApplicationRegistry` seeds ~7 apps in its constructor with a private `register(...)`; `AiProviderRegistry` registers providers in a `static{}` block with a private `register(AiProviderDescriptor)`; `DynamicChatClientProvider` wires tool beans with `@Autowired` and resolves tool groups through a hardcoded `addToolsForGroup` switch. There is no way to add an AI application, provider, or tool group without editing and redeploying the core.
- There is no plugin authorization. `OidcController` exposes the OAuth2/OIDC endpoints (`/.well-known/openid-configuration`, `/oauth2/authorize|token|userinfo|jwks`, `/oauth2/minio-sso`) but currently serves only the MinIO console SSO client; there is no client registry, no `client_credentials` flow, no consent model, and no plugin token issuance. The only auth path is the `Manager-Token` user session (UUID token in Redis via `TokenService`).
- There is no domain event bus in the business layer. `ApplicationEventPublisher` is used only in the IM module; domain events like `customer.created` / `task.updated` and any signed-webhook delivery to out-of-process consumers do not exist.
- There is no marketplace client, artifact signature verification, license model, or plugin install/lifecycle record. Custom fields, theming, and AI apps can only be changed by humans through existing UIs, not applied declaratively from a signed manifest and reversed on uninstall.
- The frontend cannot register routes, sidebar modules, themes, or chat apps dynamically. Routes are static `RouteRecordRaw` with `createWebHashHistory` (no `router.addRoute`), sidebar membership is a hardcoded key list in `src/utils/sidebarModuleOrder.ts` (only ORDER is persisted), and the chat app allow-list is a hardcoded `Set` in `src/stores/chat.ts`.

This change delivers only the **foundation** (P0 authorization + Extension Gateway + lifecycle, and P1 declarative plugins + dynamic registries + Marketplace Client). It deliberately lays the contracts (OAuth, event bus, gateway, manifest, signing) so later phases (out-of-process AI/automation runtime, billing productization, UI sandbox GA, the control-plane SaaS) slot in without rework. Third-party backend code never enters the JVM or the database — backend extension code is always out-of-process.

> **Grounding note — single-tenant reality.** This codebase is effectively **single-tenant**: `tenant_id` does not appear in `crm_init_postgres.sql`, and there is no `TenantLineInnerInterceptor`, `TenantContextHolder`, `crm_tenant` table, or `AiQuotaService`/credit system. `MybatisPlusConfig` registers only a `DataPermissionInterceptor` (→ `GlobalDataPermissionHandler`, row-level scoping by user/dept/role for `Customer`/`Contact`/`Task`/`FollowUp`/`Knowledge`) and a `PaginationInnerInterceptor`. The richer multi-tenant + billing architecture described in `CLAUDE.md` is **not present** in this open-source cut. Accordingly this proposal designs for single-tenant **per-instance** plugins, while reserving a **dual-track tenancy** seam (a nullable `tenant_id` column on the new operational tables plus an `InstanceScopeResolver` abstraction) so multi-tenancy can be added later without schema rework.

## What Changes

- Make AI capability registration dynamic by introducing a registry layer that MERGES built-in registrations with installed registrations: `ChatApplicationRegistry` keeps its constructor-seeded built-ins and additionally loads installed apps from a new `crm_ai_application` table via a new `loadInstalled(...)` merge path (built-ins win on `code` conflict); `AiProviderRegistry` gains a public `registerInstalled(AiProviderDescriptor)` path so installed providers overlay the `static{}` block without removing built-ins; `DynamicChatClientProvider` tool-group resolution consults a registry that includes installed (out-of-process) tool groups in addition to the hardcoded `addToolsForGroup` switch.
- Add a signed declarative-manifest parser (schema v1) that translates a manifest into calls on EXISTING services — custom fields via `CustomFieldServiceImpl.addField(CustomFieldAddBO)` / `DynamicSchemaServiceImpl`, theme CSS-variable token packs (`--wk-*`), `crm_ai_application` rows, and AI provider descriptors — executing zero third-party code, idempotent, and fully reversible on uninstall (`deleteField`, removed rows, deregistered descriptors).
- Add a domain event bus extending the existing Spring `ApplicationEventPublisher` pattern (e.g. `customer.created`, `task.updated`), published from business services via `@TransactionalEventListener(phase = AFTER_COMMIT)`, with a webhook dispatcher that delivers subscribed events to out-of-process plugin endpoints from `crm_plugin_subscription` using HMAC-signed payloads with retries and backoff. Dispatch failure never rolls back the business transaction.
- Add per-instance plugin lifecycle (install/enable/disable/uninstall) recorded in `crm_plugin_install` with version pinning and config stored ENCRYPTED via `SecretTextCipher`; uninstall reverses declarative extensions and revokes plugin tokens. All operational tables carry a reserved nullable `tenant_id` for the dual-track seam.
- Reuse the existing OIDC provider (`OidcController`) to add OAuth2 `client_credentials` and `authorization_code` flows for plugins; no new auth scheme is introduced. Add an instance-global `crm_oauth_client` registry of known plugin apps.
- Add a scope model that REUSES the existing `module:action` permission vocabulary enforced by `PermissionService.hasPermission(...)`: a plugin token's effective permission = (installing actor permissions) INTERSECT (granted scopes) — downscoping only, never escalation. Persist instance-admin consent in `crm_plugin_grant` before any token is issued, and store issued/rotated/revoked tokens (hashed secret only) in `crm_plugin_token`.
- Add an Extension Gateway as the single inbound API entry for plugins (base path `/plugin/gateway/**`, Bearer plugin token): validates token + scope, enforces per-plugin rate-limit/quota, writes `crm_plugin_audit_log`, and invokes an abstract `PluginAiUsageMeter` hook (no-op by default) around AI-consuming calls. Plugin data access runs the existing RBAC + row-level data-permission checks; no plugin call bypasses `PermissionService`.
- Add an instance-side Marketplace Client (fully disableable, with a local-manifest install path for air-gapped/self-host) that browses the catalog, fetches versions, downloads artifacts, and runs offline-grace license heartbeats against the abstract centralized marketplace API; it verifies every plugin version's signature against a pinned public key BEFORE install and refuses installation on verification failure. Versions are immutable; the marketplace endpoint/keys live in `SystemConfigServiceImpl`, never hardcoded under a static property.
- Add a frontend extension host: dynamic route registration (`router.addRoute`), data-driven sidebar modules reusing the `sidebarModuleOrder` persistence (`POST /managerUser/preferences` → `manager_user_ui_preferences`, V21), CSS-variable theme token registration, and a defined-but-deferred sandboxed iframe UI host contract; make the hardcoded chat-app allow-list data-driven via the existing `GET /chat/app/options` endpoint.
- Add a stable, versioned Extension API + manifest schema (v1) with documented backward-compatibility guarantees.

## Capabilities

### New Capabilities
- `plugin-extension-platform`: In-core extension framework — merge-based dynamic AI capability registries (built-in + installed), a signed declarative-manifest parser reusing `CustomFieldServiceImpl`/`DynamicSchemaServiceImpl` for data/theme/AI-app/AI-provider extensions, a domain event bus with signed-webhook dispatch, per-instance plugin lifecycle with encrypted config and a reserved `tenant_id` seam, a frontend extension host (dynamic routes/sidebar/theme + deferred iframe UI contract), and a versioned Extension API + manifest schema v1.
- `plugin-authorization`: Machine-to-machine plugin auth reusing `OidcController` for OAuth2 `client_credentials`/`authorization_code` flows, an instance-global `crm_oauth_client` registry, instance-admin consent grants, revocable hashed-secret plugin tokens, a downscoping scope model on the existing `module:action` vocabulary, and an Extension Gateway as the single inbound entry that validates token+scope, rate-limits, audits, invokes an abstract AI-usage meter hook, and preserves the existing RBAC/data-permission scoping.
- `marketplace-integration`: Instance-side Marketplace Client connecting to the abstract centralized control-plane API to browse catalog, fetch versions, download artifacts, and run license heartbeats; mandatory signature verification against a pinned public key before install; immutable version-pinned per-instance installs; online license verification with an offline grace period; and full disableability with a local-manifest install path for air-gapped/self-host.

### Modified Capabilities

None.

## Impact

- Affected backend AI packages: `backend/src/main/java/com/kakarote/ai_crm/ai/app/ChatApplicationRegistry.java`, `backend/src/main/java/com/kakarote/ai_crm/ai/app/ChatApplicationDefinition.java`, `backend/src/main/java/com/kakarote/ai_crm/ai/provider/AiProviderRegistry.java`, and `backend/src/main/java/com/kakarote/ai_crm/ai/DynamicChatClientProvider.java` (gain merge-based / public registration paths). `ManagerApplication` continues to exclude Spring AI auto-config; chat clients still come from `DynamicChatClientProvider` only.
- Affected backend auth: `backend/src/main/java/com/kakarote/ai_crm/controller/OidcController.java` and `OidcConfig` (add `client_credentials`/`authorization_code` for plugin clients), reusing `PermissionService.hasPermission(...)` for the scope vocabulary and `SecretTextCipher` for secret encryption.
- New backend packages/classes (to be created): an `InstanceScopeResolver` (dual-track seam), a declarative manifest parser, a domain event bus + signed-webhook dispatcher, a plugin lifecycle service, an Extension Gateway controller, a Marketplace Client, an abstract `PluginAiUsageMeter` hook (no-op default), plus entities/mappers/services for the new tables.
- New Flyway migration: the full reference DDL is staged at `openspec/changes/add-plugin-marketplace-platform/migration/V44__plugin_marketplace_platform.sql` (kept out of the live chain while this is a proposal). On implementation, promote it to `backend/src/main/resources/db/migration/`, renumbered to the next free version at that time (current head is `V43__im_phase3.sql`; numbers collide on merge).
- New tables (all instance-side): `crm_oauth_client` (instance-global plugin-app registry, no `tenant_id`); `crm_ai_application` (installed/custom chat apps only — built-ins stay constructor-seeded in code); `crm_plugin_install`, `crm_plugin_grant`, `crm_plugin_token`, `crm_plugin_subscription`, `crm_plugin_audit_log`. Every operational table except `crm_oauth_client` carries a **reserved nullable `tenant_id BIGINT`** (NULL = the single instance today) so a future multi-tenant interceptor can scope them without schema change. These are infra/metadata tables, not protected business entities, so `GlobalDataPermissionHandler` needs no extension.
- Reused services (no new tables for those concerns): theme/field extensions reuse `crm_custom_field` / `crm_custom_field_pool` and CSS-variable theming via `CustomFieldServiceImpl.addField`/`deleteField` and `DynamicSchemaServiceImpl.addColumn`/`dropColumn`/`columnExists`; plugin config and webhook signing secrets encrypted via `SecretTextCipher`; audit columns stamped by `MyMetaObjectHandler`; IDs from `BaseUtil.getNextId()` (Hutool Snowflake), serialized as `String` toward the frontend.
- AI-usage attribution: the gateway invokes an abstract `PluginAiUsageMeter` hook around AI-consuming calls. NOTE: this is a **non-goal to implement** — no `AiQuotaService`/credit/billing system exists in this codebase (the `CLAUDE.md` billing description is aspirational). The hook is a no-op seam so a future metering change can plug in without re-touching the gateway.
- Affected configuration: marketplace endpoint, pinned signing public key, license/heartbeat keys, and the marketplace enable/disable switch stored via `SystemConfigServiceImpl` (Redis `system:config:*`, 30-min TTL), never hardcoded under a static property; Marketplace Client is fully disableable.
- Affected frontend files: `frontend/src/router/index.ts` (add `router.addRoute` dynamic registration), `frontend/src/layouts/MainLayout.vue` + `frontend/src/utils/sidebarModuleOrder.ts` (data-driven sidebar membership reusing existing order persistence to `POST /managerUser/preferences` → `manager_user_ui_preferences`), `frontend/src/composables/useTheme.ts` + `frontend/src/styles` CSS-variable tokens (theme token registration), `frontend/src/stores/chat.ts` + `frontend/src/api/chat.ts` (make the hardcoded app allow-list data-driven via `GET /chat/app/options`), plus new plugin management views and APIs under `frontend/src/views` / `frontend/src/api`.
- Verification: backend `mvn test` (registry merge, manifest apply/reverse idempotency, OAuth `client_credentials` + scope downscoping, gateway token/scope validation, signature verification refusal, lifecycle install/uninstall reversal); frontend `npm run build` (vue-tsc type check) and `npm run lint` for the extension host and plugin management UI per AGENTS.md.
